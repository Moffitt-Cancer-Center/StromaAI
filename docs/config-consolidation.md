# Configuration Consolidation Implementation

## Overview

StromaAI deployment scripts now follow a **hierarchical configuration pattern** where variables set by one setup script are automatically available to downstream scripts through a shared global config file.

## Implementation (Commit 793b10a)

### Global Config File

**Location:** `/opt/stroma-ai/config.env`

This file serves as the single source of truth for:
- OIDC/authentication configuration
- Client secrets
- API keys
- Service URLs

### Setup Script Chain

```
setup-keycloak.sh  →  /opt/stroma-ai/config.env  →  setup-head.sh
                                                  →  setup-openwebui.sh
```

### Read by setup-head.sh (NEW)

Lines 290-307 in [deploy/head/setup-head.sh](../deploy/head/setup-head.sh):

```bash
if [[ -f "${GLOBAL_CONFIG_ENV}" ]]; then
    _oidc_discovery="$(read_config_var OIDC_DISCOVERY_URL)"
    _kc_gw_secret="$(read_config_var KC_GATEWAY_CLIENT_SECRET)"
    _kc_gw_client="$(read_config_var KC_GATEWAY_CLIENT_ID)"
    
    if [[ -n "${_oidc_discovery}" && -n "${_kc_gw_secret}" ]]; then
        log_ok "OIDC configuration found in ${GLOBAL_CONFIG_ENV}"
        export OIDC_DISCOVERY_URL="${_oidc_discovery}"
        export KC_GATEWAY_CLIENT_SECRET="${_kc_gw_secret}"
        export KC_GATEWAY_CLIENT_ID="${_kc_gw_client:-stroma-gateway}"
        _has_oidc_from_global=1
    fi
fi
```

## Deployment Workflow

### Before Implementation

Each setup script prompted independently:

```bash
./deploy/keycloak/setup-keycloak.sh
# Generate KC_GATEWAY_CLIENT_SECRET=abc123...

./deploy/head/setup-head.sh
# Prompt: "Enter OIDC discovery URL: _____"
# Prompt: "Enter gateway client secret: _____"  # Must manually re-enter abc123

./deploy/openwebui/setup-openwebui.sh
# Reads from global config ✓ (already implemented)
```

**Problem:** Manual duplication, risk of mismatched secrets

### After Implementation

Sequential execution with automatic propagation:

```bash
./deploy/keycloak/setup-keycloak.sh
# Writes OIDC_DISCOVERY_URL, KC_GATEWAY_CLIENT_SECRET to /opt/stroma-ai/config.env

./deploy/head/setup-head.sh
# Reads OIDC_DISCOVERY_URL, KC_GATEWAY_CLIENT_SECRET from /opt/stroma-ai/config.env
# No prompts needed ✓

./deploy/openwebui/setup-openwebui.sh
# Reads KC_OPENWEBUI_CLIENT_SECRET from /opt/stroma-ai/config.env
# No prompts needed ✓
```

**Benefit:** Zero duplication, guaranteed consistency

## File Structure

```
/opt/stroma-ai/
├── config.env                  # Global config (shared by all)
└── deploy/
    ├── keycloak/
    │   ├── setup-keycloak.sh   # Writes to global config
    │   └── .env                # Keycloak compose env (internal secrets only)
    ├── head/
    │   ├── setup-head.sh       # Reads from global config
    │   └── .env                # Head compose env (duplicates from global)
    └── openwebui/
        ├── setup-openwebui.sh  # Reads from global config
        └── .env                # OpenWebUI compose env (duplicates from global)
```

## Dev Environment

The dev environment (`dev/dev.sh`) remains **independent** with its own `.env` file:

- **dev/.env**: Generated by `dev/dev.sh` with separate secrets
- Does NOT read from `/opt/stroma-ai/config.env`
- Maintains isolation for local development

This is intentional - dev and production should have different:
- Ports (avoid conflicts)
- API keys (avoid cross-contamination)
- Endpoints (localhost vs production hosts)

## Backward Compatibility

The implementation maintains backward compatibility:

1. **If global config doesn't exist:** setup-head.sh falls back to prompting (same as before)
2. **If global config exists but incomplete:** Warns and prompts for missing values
3. **If global config is complete:** Uses values automatically (new behavior)

No breaking changes - existing deployments continue working.

## Questions Answered

**Q: Are variables duplicated across .env files?**  
A: Yes - global config variables are duplicated into compose-specific `.env` files for podman-compose. This is by design (compose needs its own env file).

**Q: Can one setup script update another's .env?**  
A: No - we use the **intermediate global config** pattern instead. Scripts write to global config, read from global config, then write to their own compose `.env`. This maintains loose coupling.

**Q: Should dev and prod share config?**  
A: No - dev environment remains isolated with different ports, secrets, and endpoints for safety.

**Q: What if I have multiple environments (staging, prod)?**  
A: Current implementation supports one global config per installation directory. For multiple environments, use separate installation directories (e.g., `/opt/stroma-ai-staging`, `/opt/stroma-ai-prod`), each with its own `config.env`.
